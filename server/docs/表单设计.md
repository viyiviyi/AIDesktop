# 表单交互 — 基于 PI-AI 的设计

## 系统架构关系

```
PI-AI (pi-ai 包) → 提供 streamSimple、Model、Api 类型
PI-Agent-Core (pi-agent-core 包) → 提供 Agent、AgentTool 接口
AIDesktop:
  - pi-agent-session.ts: 封装 Agent, 跑 runAgentAsync
  - pi-tools.ts: 把 MCP 服务转为 AgentTool
  - mcp/service.ts: MCP 服务注册表, 实现具体逻辑
  - services/eventBus.ts: 广播事件, 前端通过 WS 接收
  - services/conversation.ts: 持久化会话
```

## 设计要点

### 1. MCP 服务方式

新增一个 MCP 服务 `mcp.form`，只有一个方法 `requestInput`。这是**服务端**的工具，AI 调用时和调用 `mcp.filesystem.read` 等没有区别，走相同的执行路径。

### 2. 执行流程

AI 调用 `mcp.form.requestInput({ prompt, schema })`：

- 服务端收到调用后，返回给 pi-agent-core（toolResult）一个标志 `{ status: 'pending', formId: 'xxx' }`
- 同时通过 eventBus 发送 `form_request` 事件 → WS → 前端
- 注意：**服务端不等待**，直接结束工具调用。表单的提交/取消是后续通过 REST API

前端收到 `form_request` 事件：
- 在消息列表中工具调用卡片下方渲染一个可填写表单
- 在通知中心添加通知
- 锁定输入区

用户提交/取消表单：
- 调用 REST API `POST /api/apps/:appId/conversations/:convId/form-response`
- 服务端：
  - 在当前会话中插入一条 toolResult 消息（role='toolResult'，对应表单的工具调用 ID）
  - 保存会话
  - 恢复 agent 执行：调用 `runAgentAsync` 继续处理

### 3. 关键—让 AI 停下来

需求说：AI 调用表单工具后不生成后续回复，等用户提交。

这在 PI-AI 体系下的实现方式：

**方法 A：工具返回 "pending" 标记，Agent 工具执行函数返回一个特殊值**

在 `pi-tools.ts` 的 `execute` 中，如果检测到 form 请求，不等待用户，直接返回 `{ status: 'pending', formId }`。但 pi-agent-core 拿到这个 toolResult 后，仍然会继续调用模型，模型会继续生成回复，无法停止。

**方法 A 有问题**，因为 agent 不知道"等待用户输入"这个概念。

**方法 B（推荐）：无需 agent 参与等待**

核心思路：表单工具调用后，agent 确实会继续——但 agent 生成的最终消息先不发给模型作为下一轮的上下文。session 的保存和 agent 的继续执行延迟到用户提交表单后。

具体流程：

1. AI 调用 `mcp.form.requestInput`：
   - toolResult 立即返回 `{ status: 'pending', formId: 'xxx' }`
   - agent 看到 toolResult = "pending"，可以生成一条消息如"请填写以下表单..."，然后会进入下一轮 thinking 等待更多输入，或者因为没更多输入而结束 round
   - **关键点**：此时服务端向 eventBus 发 `form_request` 事件，但**不保存 agent 的后续消息**

2. 服务端保存会话状态标记 `status: 'form-pending'` 到会话中

3. 前端显示表单，用户填写并提交

4. 用户提交后调用 REST API：
   - 服务端插入 toolResult 消息（内容 = 用户填的表单数据）
   - 重新调用 `runAgentAsync`（带已有历史 + 新插入的 toolResult）
   - agent 继续运行，看到表单数据作为工具结果，正常处理

**关键问题**：第 1 步中 AI 调用表单后，agent 仍在运行。我们需要的效果是——agent 在第 1 步的工具执行结束后**尽快停止**。

**方法 C（最简洁的解法）：让工具执行抛出一个可控的"暂停"异常**

在 `mcp.form.requestInput` 的实现中，发完 eventBus 后抛出一个特殊错误。pi-tools.ts 的 `execute` 捕获到这个错误，返回 `isError: true` 且内容为 `{ status: 'paused', formId: 'xxx' }`。

但这样 agent 仍会继续——它会看到错误结果后尝试重试或换方法。

**方法 D（最终方案）：暂停 + eventBus 驱动恢复**

在 `pi-tools.ts` 和 `pi-agent-session.ts` 中增加对 form 工具的特殊处理路径：

**pi-tools.ts** 中 `mcp.form` 的 execute:
- 调用 `mcpServiceRegistry.callMethod('mcp.form', 'requestInput', ...)`
- 这个方法不会返回，而是：
  1. 生成 formId
  2. 发 `form_request` event（带 form schema）
  3. 保存会话（当前消息 + 工具调用标记）
  4. **挂起 agent**（通过一个 Promise 等待外部触发）
- 当用户提交表单后，REST API handler 解析 Promise，返回表单数据

但这和 pi-agent-core 的流式架构冲突——execute 是同步工具调用，不能长时间挂起。

**方法 E（推荐的最终方案）：两步走——工具调用只负责发送表单给前端，不等待**

1. AI 调用 `mcp.form.requestInput` → 工具执行：
   - 发 event (form_request) 给前端
   - 返回 toolResult: `{ status: 'pending', formId, message: '请用户填写表单' }`
   - agent 获得这个结果，生成一条消息如"我需要你填写一些信息..."
   - agent 进入下一轮，调用方（pi-agent-session.ts 的 runAgentAsync）检测 agent 状态

2. runAgentAsync 中，每次 agent 一轮执行完成后：
   - 检查是否有 pending 的 form（在会话对象上有标记）
   - 如果有，保存当前消息但**不发射 done 事件**
   - 保存会话，设置 conv.pendingForm = formId
   - 等待 eventBus 上 `form_response` 事件

3. 用户提交表单 → REST API：
   - 查找 conv.pendingForm
   - 将用户数据作为一条 toolResult 手动插入到 agent 的 state.messages
   - 调用 `agent.prompt()` 继续执行（或者重新走 runAgentAsync 流程）

这个方案的好处：**不需要改造 pi-agent-core**。

### 4. 状态存储

在 Conversation 模型中增加字段：

```typescript
interface Conversation {
  // ...现有字段
  status?: 'active' | 'form-pending';
  pendingForm?: {
    formId: string;
    toolCallId: string;
    schema: FormSchema;       // 表单定义
    createdAt: string;
  } | null;
}
```

### 5. EventBus 新增事件

```typescript
type ConvEventType =
  | ...现有事件
  | 'form_request'     // AI 请求用户填写表单
  | 'form_response'    // 用户提交了表单
  | 'form_cancelled';  // 用户取消了表单
```

### 6. 前端表单渲染

`form_request` 事件携带：
```json
{
  "type": "form_request",
  "appId": "...",
  "convId": "...",
  "data": {
    "formId": "form-xxx",
    "toolCallId": "tc-xxx",
    "schema": {
      "title": "基本信息",
      "description": "请输入以下信息",
      "fields": [
        { "name": "name", "label": "姓名", "type": "text", "required": true },
        { "name": "age", "label": "年龄", "type": "number" },
        { "name": "gender", "label": "性别", "type": "select", "options": ["男", "女"], "required": true },
        { "name": "confirm", "label": "确认提交", "type": "confirm" },
        { "name": "file", "label": "上传文件", "type": "file", "accept": ".pdf,.doc" }
      ]
    },
    "conversationId": "conv-xxx",
    "appId": "app-xxx",
    "createdAt": "..."
  }
}
```

### 7. 前端实现

1. **消息列表渲染**：工具调用卡片（已有的 tool-call-item）下面，如果这个工具调用所对应的 form 还在 pending，渲染一个可交互的表单。

2. **通知中心**：一个浮动的通知面板，显示所有 pending 表单。点击进入某个表单时，滚动到对应消息位置。

3. **输入区锁定**：
   - 新的 state: `pendingForms: Map<string, FormSchema>`（key 是 formId）
   - 当 pendingForms.size > 0：禁用 textarea 和发送按钮，显示"请先填写并提交上方的表单"
   - 当 pendingForms.size === 0：恢复正常

4. **表单提交**：`POST /api/apps/:appId/conversations/:convId/form-response`，body: `{ formId, data: { fieldName: value } }`

5. **表单取消**：提交 formId + `{ cancelled: true }`

6. **刷新恢复**：
   - 首次加载时，如果 conv.status === 'form-pending':
     - 重新渲染表单（从 conv.pendingForm.schema 读取）
     - 锁定输入区
     - 不重新触发 form_request 事件（因为表单已经在消息列表中展示了）

### 8. 刷新/重启后恢复

服务端启动时扫描所有会话：
- 对于 status === 'form-pending' 的会话
- 前端打开该窗口和会话时，检测到 pendingForm
- 自动恢复表单渲染

### 9. 多次表单

一个会话可以有多个等待表单：
```typescript
pendingForms: FormPendingInfo[] = [];
```
每个表单提交/取消后移除。当数组为空时输入区恢复可用。

### 10. AI 侧表单定义

在 `mcp.form.requestInput` 的参数中，schema 由 AI 动态生成：

```
调用: mcp.form.requestInput({
  title: "用户信息",
  description: "请填写以下信息完成注册",
  fields: [
    { name: "username", label: "用户名", type: "text", required: true },
    { name: "age", label: "年龄", type: "number" },
  ]
})

返回: { status: "pending", formId: "form-xxx", message: "已发送表单给用户" }

// 用户提交后，toolResult 自动作为上下文
// AI 看到的数据: { name: "张三", age: 25 }
```

### 11. 实现步骤

1. **服务端**: 在 mcp/service.ts 新增 `mcp.form` 服务(只有 requestInput 方法)
2. **服务端**: Conversation 模型增加 status/pendingForm 字段
3. **服务端**: eventBus 新增 form_request/form_response 事件类型
4. **服务端**: REST API: POST /apps/:appId/conversations/:convId/form-response
5. **服务端**: pi-agent-session.ts 在 runAgentAsync 中检测 pendingForm 并暂停
6. **服务端**: form-response handler 插入 toolResult 到 session 并恢复 agent
7. **前端**: api.ts 新增 submitFormResponse 方法
8. **前端**: Window.tsx 新增表单组件，消息列表中的表单渲染
9. **前端**: Window.tsx 通知中心和输入区锁定逻辑
10. **前端**: 页面加载时恢复 pending 表单状态

### 12. 字段类型

| type | UI 控件 | 数据结构 |
|------|---------|---------|
| text | 单行文本框 | string |
| textarea | 多行文本框 | string |
| number | 数字输入框 | number |
| select | 下拉选择框 | string \| null |
| radio | 单选框 | string \| null |
| checkbox | 复选框 | boolean |
| confirm | 确认按钮 | boolean (true=确认) |
| file | 文件选择器 | string (base64 或文件路径) |

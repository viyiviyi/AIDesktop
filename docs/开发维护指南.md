## 1. 项目概述

AIDesktop 是一个运行在浏览器中的 AI 桌面环境。用户看到的仿 macOS 桌面操作系统，背后是一个多 Agent 协作系统——每个桌面"应用"都是一个独立的 AI Agent。

### 1.1 核心概念

| 概念 | 说明 |
|------|------|
| **App（应用）** | 一个 AI Agent，拥有独立的 system prompt、工具集和会话历史 |
| **Agent（智能体）** | 基于 pi-agent-core 的 AI 会话引擎，可独立运行和互相调用 |
| **MCP 服务** | 工具扩展接口，支持 Stdio/SSE/HTTP 三种传输协议 |
| **Skill（技能）** | 可插拔的知识模块，为 Agent 提供专业领域能力 |
| **Conversation（会话）** | 用户与 Agent 的对话记录，存储在文件系统 |
| **Desktop（桌面）** | 仿 macOS 的 UI 框架，管理窗口、Dock、开始菜单 |

## 4. 架构核心

### 4.1 前后端通信

```
浏览器 (React SPA)          Express 后端
      │                         │
      ├── REST API ───────────► │  /api/apps, /api/settings, ...
      │                         │
      ├── SSE (legacy) ───────► │  /api/chat/:appId/:convId
      │                         │
      └── WebSocket ◄─────────► │  /api/ws  (新方案，实时事件推送)
                                │
      Agent 事件流:              │
      thinking -> text_chunk    │
      -> tool_call -> tool_result -> done
```

**当前状态**（2025-06）：WebSocket 是主通道，前端通过 `useAgentEventStream` hook 连接并订阅 convId 的事件。

### 4.2 AI Agent 调用流程

```
用户发送消息
  │
  ▼
routes/conversations.ts
  │ POST /:appId/:convId/chat
  │
  ▼
agentEngine.processMessage(appId, convId, userContent)
  │ 1. 保存 user 消息到会话文件
  │ 2. 获取/创建 PiAgentSession（单例，按 appId）
  │ 3. session.syncHistory(历史消息) —— 同步到 pi-agent 状态
  │ 4. session.sendMessage(convId, history, userContent)
  │    ├── 内部调用 agent.prompt(userText)
  │    ├── pi-agent 根据 system prompt + 历史 + 工具列表，调用 LLM
  │    ├── LLM 可能触发工具调用（mcp.xxx）
  │    └── 工具结果返回后继续推理（循环，直到输出最终回复）
  │ 5. 保存 assistant 回复到会话文件
  │
  ▼
返回 { assistantMessage }
```

**后台异步模式**（Agent 间调用时使用）：
```
runAgentAsync() —— 通过 EventBus 推送到 WebSocket
  │ 前端实时接收 thinking / text_chunk / tool_call 等事件
```

### 4.3 数据分离模式

```
app 定义层（不可变）：               用户配置层（运行时修改）：
apps/{source}/{id}/                apps_data/{id}/
  meta.json  —— app 定义/默认值       config.json —— 用户运行时覆盖
  app.md     —— system prompt        app.md      —— 用户自定义 prompt（覆盖）
  mcp.json   —— 关联的 MCP 服务列表
  skills/    —— 技能配置
```

**合并规则**：运行时读取时，`config.json` 中有定义的字段覆盖 `meta.json` 中的默认值（见 `apps.ts` 中的 `mergeConfig` 函数）。

## 5. 关键数据模型

### 5.1 Message 消息模型

```typescript
// 消息角色
type MessageRole = 'user' | 'assistant' | 'system' | 'toolResult';

// 消息体
interface Message {
  id: string;
  role: MessageRole;
  content: Content[];           // 内容块数组（支持多模态）
  timestamp: string;
  toolCalls?: ToolCall[];       // assistant 消息中的工具调用列表
  toolResultMeta?: ToolResultMeta; // toolResult 消息中的元信息
  replyTo?: string;             // 回复链（编辑功能）
  edited?: boolean;             // 编辑标记（旧消息标记为 edited，新分支消息插入）
}

// 内容块
type Content =
  | { type: 'text'; text: string }
  | { type: 'image'; url: string; alt?: string }
  | { type: 'file'; path: string; name: string; size: number }
  | { type: 'toolCall'; id: string; name: string; arguments: {} }
  | { type: 'tool_result'; toolCallId: string; toolName: string; result: unknown; isError: boolean }
  | { type: 'thinking'; text: string };
```

### 5.2 Conversation 会话模型

```typescript
interface Conversation {
  id: string;
  appId: string;
  title: string;
  createdAt: string;
  updatedAt: string;
  messages: Message[];
  source?: 'user' | 'agent' | 'system';
  callChain?: Array<{ callerAppId: string; callerConvId?: string; callId?: string; timestamp: string }>;
  pendingUserInput?: string | null;
  status?: 'active' | 'form-pending';
  pendingForms?: Array<{ formId: string; toolCallId: string; schema: FormSchema; createdAt: string }>;
  workspaceDir?: string | null;       // 会话工作目录（绝对路径）
  authorizedDirs?: string[];          // 已授权的目录列表
}
```

**存储格式**：每个会话一个文件夹 `apps_data/{appId}/conversations/{yyyyMMddHHmmss}/conversation.json`

### 5.3 Agent 间调用机制 (mcp.agent.call)

```
Agent A ──mcp.agent.call({ agentId: "B", message: "帮我..." })──► Agent B
  │                                                                 │
  │ 1. 创建新会话（标记 source=agent）                               │
  │ 2. 继承调用者的 workspaceDir                                     │
  │ 3. runAgentAsync 异步处理                                       │
  │ 4. 通过 EventBus 事件等待结果                                    │
  │ 5. 结果返回给 Agent A                                            │
  └─────────────────── 结果 ◄─────────────────────────────────────────┘
```

**可见性控制**：通过 `visibleApps`（可见的桌面应用）和 `visibleServices`（可见的后台服务）控制 Agent 间可见性。

### 6.2 状态管理

采用 `React Context + useReducer` 模式，单一 `DesktopState` 包含：

```typescript
interface DesktopState {
  settings: DesktopSettings;           // 桌面设置
  installedApps: AppInfo[];            // 安装的应用列表
  windows: WindowState[];              // 打开的窗口
  focusedWindowId: string | null;      // 当前焦点窗口
  startMenuOpen: boolean;              // 开始菜单状态
  startMenuMode: 'click' | 'voice';   // 开始菜单模式
  taskbarApps: string[];               // 运行中的应用 ID
  appLastPositions: Record<string, { x: number; y: number }>; // 窗口位置记忆
  conversationTitles: Record<string, string>; // 会话标题缓存
}
```

**为什么不用 Redux/Zustand**：状态结构相对简单，Context + useReducer 足够满足需求，减少依赖。

### 6.3 窗口管理

- 窗口支持：拖拽移动、调整大小、最小化、最大化、层叠
- 窗口位置记忆：同应用关闭后记住上次位置（通过 `appLastPositions`）
- 同应用多开：每多开一个偏移 30px
- 通讯：`Window.tsx` 通过 props 接收 appId，内部根据 appId 调用对应 API

## 7. 后端关键流程

### 7.1 应用加载流程

```
服务器启动
  │
  ▼
index.ts -> init()
  │
  ├── ensureDir() // 确保目录结构
  │
  ├── appLoader.loadAll()
  │   ├── loadFromDirectory(SYSTEM_APPS_DIR, 'system')
  │   ├── loadFromDirectory(USER_APPS_DIR, 'user')
  │   ├── loadFromDirectory(MARKETPLACE_APPS_DIR, 'marketplace')
  │   └── 每个 app 读取: meta.json + app.md + mcp.json + skills + config.json
  │
  ├── settingsService.getSettings()
  │
  └── mcpClientRegistry.initializeFromConfig()
      └── 根据 configs 中的 MCP 配置启动外部连接（子进程/SSE/HTTP）
```

**重要**：应用按 `system > user > marketplace` 优先级加载，同 id 的应用只有第一个生效。

### 7.2 消息处理管道

用户消息到达后：
1. **AgentEngine.processMessage()** — 入口
2. **conversationService.addMessage()** — 保存用户消息到磁盘
3. **piAgentManager.getOrCreate()** — 获取/创建会话（每个 appId 一个单例 Session）
4. **session.syncHistory()** — 同步历史到 pi-agent（过滤 edited 标记）
5. **agent.prompt()** — 调用 LLM（可能触发多轮工具调用）
6. **conversationService.addMessage()** — 保存 assistant 回复
7. 返回结果

### 7.3 消息编辑（回退点）

```
用户编辑一条消息
  │
  ▼
PUT /:appId/:convId/messages/:msgId
  │
  ├── 原消息标记 edited: true（保留在原位）
  ├── 在原消息之后插入新消息（同 role，新 id）
  └── syncHistory 时过滤 edited 消息
```

这样做的优势：
- 保留原始对话痕迹（不破坏历史）
- 支持回溯到任意编辑版本
- 实现简单，无需管理分支树

### 7.4 WebSocket 事件通道

```
Agent 事件
  │
  ▼
EventBus.emit()
  │
  ├── 通知该 convId 的订阅者（用于 agent 间调用 await）
  │
  └── 通知 '__all__' 订阅者
       │
       ▼
      wsServer → 广播给所有订阅了该 convId 的 WebSocket 客户端
```

事件类型：
- `thinking` — Agent 思考中
- `message_start` / `message_update` — 消息开始/更新
- `text_chunk` — 文本流式输出
- `tool_call` — 工具调用
- `tool_result` — 工具结果
- `user_input_request` — 等待用户输入
- `agent_call_start` / `agent_call_end` — Agent 间调用

### 7.5 表单交互流程

```
Agent 调用 mcp.form.requestInput
  │
  ▼
EventBus.emit('form_request')
  │ 前端 WebSocket 收到，展示表单 UI
  │
  ▼
用户填写并提交
  │
  ▼
POST /:appId/:convId/form-response
  │
  ├── EventBus.emit('form_response')
  ├── handleFormPendingInternal() 准备 toolResult
  └── runAgentAsync() 恢复 Agent
```

## 8. 内置 MCP 服务

| 服务名 | 类别 | 功能 |
|--------|------|------|
| `mcp.window` | admin | 窗口管理（打开/关闭/聚焦/最小化/最大化） |
| `mcp.filesystem` | admin | 文件系统（文件读/写/搜索/替换/列表） |
| `mcp.settings` | admin | 系统设置（获取/更新设置、技能管理、应用管理） |
| `mcp.agent` | builtin | Agent 管理（列出 Agent、获取信息、调用 Agent、请求用户输入） |
| `mcp.sleep` | builtin | 等待指定秒数 |
| `mcp.exec` | builtin | 执行 shell 命令 |
| `mcp.http` | builtin | HTTP 请求 |
| `mcp.browser` | builtin | 浏览器控制（Playwright 驱动） |
| `mcp.form` | builtin | 表单交互（向用户展示表单） |
| `mcp.skill` | builtin | 技能服务（列出/读取/执行技能） |
| `workspace.code` | workspace | 工作区文件编辑（路径相对于会话工作目录） |
| `workspace.dir` | workspace | 工作目录管理 |

所有内置服务在 `mcp/service.ts` 中通过 `switch-case` 路由到处理函数。外部 MCP 连接（如 Playwright MCP Server）通过 `clientRegistry.ts` 管理。

## 9. 开发任务指南

### 9.1 本地开发环境

```bash
# 启动后端（热重载）
cd server && npm run dev    # tsx watch src/index.ts

# 启动前端（热重载）
cd client && npm run dev    # vite

# 访问 http://localhost:5173（Vite 代理到 27135）
```

### 9.2 新增一个 API 路由

1. 在 `server/src/routes/` 下创建新的路由文件
2. 在 `server/src/index.ts` 中 `app.use('/api/xxx', xxxRouter)`
3. 类型定义在 `server/src/types/index.ts` 中

### 9.3 新增一个前端页面/窗口

1. 在 `client/src/components/` 创建新组件
2. 如果是新应用类型，在 `Window.tsx` 中添加对应的内容渲染分支
3. API 调用封装在 `client/src/services/api.ts` 中

### 9.4 新增一个内置 MCP 工具

1. 在 `mcp/service.ts` 的 `builtInServices` 中添加服务定义
2. 在 `callMethod` 的 switch-case 中添加路由分支
3. 实现对应的 `handleXxxMethod` 方法

### 9.5 新增一个外部 MCP 连接

1. 在 `desktop_data/configs/modes.json` 的 MCP 配置中添加连接
2. 支持三种传输类型：stdio / sse / http
3. 重启后端或调用 `/api/mcp/reload` 重新加载

### 9.6 创建新应用

两种方式：
- **UI 方式**：通过开始菜单中的"创建应用"功能对话式创建
- **手动方式**：
  1. 在 `desktop_data/apps/system/` 或 `user/` 下创建 app 目录
  2. 创建 `meta.json`（应用元数据）
  3. 创建 `app.md`（system prompt）
  4. 创建 `mcp.json`（关联的 MCP 服务列表）
  5. 创建 `skills/` 目录（技能配置）
  6. 调用 POST `/api/apps/reload` 重新加载

### 9.7 构建发布版本

```bash
./scripts/build-dist.sh        # release 模式
./scripts/build-dist.sh debug  # debug 模式（保留 node_modules）

# 输出在 build/aidesktop/，运行 start.sh 或 start.bat
```

### 9.8 模型提供商配置

编辑 `desktop_data/configs/modes.json`：

```json
{
  "providers": [
    {
      "id": "deepseek",
      "name": "DeepSeek",
      "apiType": "openai",
      "apiKey": "sk-xxx",
      "baseUrl": "https://api.deepseek.com/v1",
      "enabled": true,
      "models": [
        { "id": "deepseek-chat", "name": "DeepSeek Chat", "maxTokens": 8192, "supports": ["text"], "params": {} }
      ]
    }
  ]
}
```

支持的类型：`openai`（兼容）、`anthropic`、`custom`

### 9.9 查找日志

- **后端日志**：写入 `desktop_data/logs/` 目录，通过前端 LogPanel（`/api/logs`）可查看
- **运行日志**：server 终端输出（serverLogger）
- **前端日志**：浏览器控制台，或前端 LogPanel

## 10. 常见维护事项

### 10.1 依赖更新

- 前端依赖：`cd client && npm update`
- 后端依赖：`cd server && npm update`
- pi-ai / pi-agent-core 是本地 vendor（`vendor/` 目录），需手动同步

### 10.2 数据迁移

会话数据有两种存储格式共存：
- **旧格式**：`{uuid}.json`（单个 JSON 文件）
- **新格式**：`{yyyyMMddHHmmss}/conversation.json`（文件夹 + 附件支持）

加载时自动从旧格式迁移到新格式（`conversation.ts` 的 `loadConversations` 中有迁移逻辑）。

### 10.3 会话附件

附件（图片等）存储在会话文件夹中，与 `conversation.json` 同级。

### 10.4 安全性

- API Key 存储在 `modes.json` 中（本地文件系统，不暴露到前端）
- 应用可见性通过 `visibleApps` / `visibleServices` 控制
- 文件系统工具限制在 `public_data` 目录内（路径穿越检测）
- 工作目录系统通过授权机制控制

### 10.5 退出文档未覆盖的内容

以下文件和目录值得关注：
- `vendor/pi/` — pi-ai 和 pi-agent-core 的本地源码，定义 Agent 的核心行为
- `docs/plans/` — 已废弃/待实现的计划文档
- `.claude/mcp.json` — Claude 的 MCP 配置
- `server/src/models/hermes.ts` — Hermes API 适配器
